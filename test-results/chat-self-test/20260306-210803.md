## Phase 6 — Issue Analysis

### Bug #1: GET /api/sessions/:id/messages 503 Storm in Relay Mode

**Observed:** During a Relay mode session, the client sends 20+ GET requests to `/api/sessions/{agent-id}/messages?cwd=...` in quick succession. Most return 503; only 1 eventually succeeds (200). Requests begin before the first message is sent and continue after streaming completes.

**Expected:** Either (a) the endpoint returns 200 with `{ messages: [] }` consistently while the session is in Relay mode, or (b) polling is suppressed/disabled in Relay mode since history is managed differently.

**Root cause analysis:**
- `GET /messages` route (`sessions.ts:116-140`) has no explicit 503 path — it delegates to `runtime.getMessageHistory()` → `transcriptReader.readTranscript()`, which returns `[]` on missing file
- Server error handler (`middleware/error-handler.ts`) returns 500, not 503
- The only 503 paths in the server are: SSE connection limits (`session-broadcaster.ts:96,103`) and binding subsystem (`relay.ts:483+`)
- **Most likely cause**: The route handler is `async` with no `try-catch`. If `validateBoundary` or any upstream call throws, the unhandled rejection may cause the Vite proxy to time out and return 503 to the client
- **Secondary cause**: The client polls every 3 seconds (`refetchInterval` in `useChatSession.ts`) with `staleTime: 0`, creating high-frequency traffic regardless of Relay mode state

**Code locations:**
- `apps/client/src/layers/features/chat/useChatSession.ts` — `refetchInterval` + `MESSAGE_STALE_TIME_MS = 0`
- `apps/server/src/routes/sessions.ts:116` — no try-catch on async handler
- `apps/server/src/services/runtimes/claude-code/transcript-reader.ts:289` — `validateBoundary(vaultRoot)` may throw

**Recommendation:**
1. Add `try-catch` wrapper to GET /messages route handler
2. Disable the polling refetch in Relay mode (rely on `sync_update` SSE events instead)
3. Increase `MESSAGE_STALE_TIME_MS` from 0 to 30,000ms

### Bug #2: Message Duplication for Tool-Call Responses in Relay Streaming Mode

**Observed:** When an assistant response includes tool calls (e.g., TodoWrite, Skill), the final text response after the tool call renders twice in the chat UI during live streaming. The exact same text block, with the same timestamp, appears as two separate message bubbles.

**Expected:** Each assistant response appears exactly once.

**Not affected:** History reload renders correctly (single occurrence). Text-only responses without tool calls render once.

**Root cause (from code trace):**

The duplication is a race condition between two concurrent update paths in `useChatSession.ts`:

1. **Streaming path**: `relay_message` SSE events → `streamEventHandler()` → `updateAssistantMessage()` → local messages state updated with `text_delta` chunks and the final text

2. **History sync path**: After the SDK writes the complete message to JSONL, `SessionBroadcaster` detects the file change (chokidar) → fires `sync_update` SSE event → client handler: `queryClient.invalidateQueries(['messages', sessionId, cwd])` → `historyQuery` refetches → `history.slice(currentMessages.length)` logic runs

**The race**: The file write and chokidar `sync_update` event can arrive while the final `text_delta` events from the relay_message stream are still being processed OR right as `done` fires (setting status to 'idle'). At this moment, if the history refetch returns the just-written message, the history merge logic:
```typescript
if (historySeededRef.current && !isStreaming) {
  const newMessages = history.slice(currentMessages.length);
  if (newMessages.length > 0) setMessages(prev => [...prev, ...newMessages]);
}
```
...sees the final text response as a "new" message (not yet counted in `currentMessages.length`) and appends it again, even though it's already been added via the streaming path.

**Why tool calls make it worse**: With tool calls, the streaming pipeline has more events to process (tool_use, tool_result, then final text). The file write happens earlier relative to when the final text arrives in the streaming pipeline, widening the race window.

**Why history load is clean**: No concurrent streaming state — history is the single source of truth.

**Code locations:**
- `apps/client/src/layers/features/chat/useChatSession.ts` — `sync_update` handler (invalidateQueries) + history merge effect + `isStreaming` guard
- `apps/server/src/services/runtimes/claude-code/session-broadcaster.ts` — file watcher → `sync_update` emission
- `apps/server/src/services/runtimes/claude-code/sdk-event-mapper.ts` — event ordering after tool calls

**Recommended fix:**
```typescript
// In useChatSession.ts sync_update handler:
eventSource.addEventListener('sync_update', () => {
  if (isStreaming) return; // Skip invalidation during active streaming
  queryClient.invalidateQueries({ queryKey: ['messages', sessionId, selectedCwdRef.current] });
});
```
Or alternatively: deduplicate by message ID in the history merge step rather than slicing by count.

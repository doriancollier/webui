## Phase 6 — Issue Analysis & Deep Research

### Bug 1: streamdown v2.3.0 truncates `[]` in inline code during streaming

**Class:** Bug
**Research:**
- Affected file: `apps/client/src/layers/features/chat/ui/StreamingText.tsx`
- Library: `streamdown@2.3.0` (pinned as `"latest"` in `apps/client/package.json`)
- `StreamingText` passes the full accumulated text to `<Streamdown>` on every render. The text is correct (confirmed in `stream-event-handler.ts:134-137`). The bug is in streamdown's incremental parser.
- Confirmed streaming-path only: history reload renders `number[]` and closing sentence correctly.
- Root cause: streamdown's incremental renderer enters an error state when `[]` appears at a chunk boundary inside a backtick span. The `[` is interpreted as the start of a Markdown link reference syntax `[label][ref]`, consuming subsequent content until it either finds a closing `]` or discards the remainder of the block.
- Reproducible pattern: `**Bold**: \`code[]\`` where `[]` is the final token of a streamed chunk.
- Fix direction: Pin streamdown to a newer version (if fixed upstream), or apply a workaround in `StreamingText.tsx` (e.g., append a zero-width space or sentinel to prevent boundary-state misparsing).

### Bug 2: 503 storm — `GET /messages` polled with Agent-ID in relay mode

**Class:** Bug (pre-existing, confirmed in `research/20260307_relay_streaming_bugs_tanstack_query.md`)
**Research:**
- `useChatSession` queries `transport.getMessages(sessionId, ...)` where `sessionId` comes from the URL param `?session=`.
- In relay mode, `?session=` holds the Agent-ID (e.g., `c2392cb8`), not the SDK session ID.
- `GET /api/sessions/c2392cb8.../messages` returns 503 because no JSONL file exists named `c2392cb8.jsonl` (the JSONL is named `e8623b30.jsonl`).
- `refetchInterval` in `historyQuery` does not check `relayEnabled` — it polls even while streaming and even though every poll returns 503.
- Network log confirmed: 299 requests total, majority 503 on this endpoint.
- Fix direction (from research doc): Gate `refetchInterval` on `!relayEnabled`; use SDK session ID from sidebar/session metadata, not URL param for history fetches.

### Bug 3: Missing messages after navigate-away/back in relay mode

**Class:** Bug
**Research:**
- Affected file: `apps/client/src/layers/features/chat/model/use-chat-session.ts:210-216`
- After navigate-away/back, session ID changes from Agent-ID (`c2392cb8`) to SDK session ID (`e8623b30`) because sidebar navigation uses the session's canonical SDK ID.
- The history seeding logic at line 212 uses `history.slice(currentMessages.length)` to append only new messages on polling updates. But when relay SSE reconnects and adds messages to state before a polling refetch, `currentMessages.length` > 0, causing the subsequent `setMessages` call to skip the already-present messages and only append the tail.
- Effectively: the initial seed correctly loads all 13 messages, but relay SSE event replay overwrites state with a partial set, then the next poll can only append the tail.
- Observable result: 8 of 13 messages visible; first 2 full exchanges absent.
- Fix direction: Replace `slice(currentMessages.length)` with a deduplication strategy (e.g., merge by message ID, or always replace the full set when not streaming). The incremental append pattern is fundamentally fragile when multiple state sources (history query + relay SSE) update the same state.

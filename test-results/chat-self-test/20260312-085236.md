## Issues Found

### Bug #1 — Duplicate Message Rendering During Live Streaming (CRITICAL)
**Observed:** Every assistant response appears twice in the UI during live streaming: once before the user's message bubble, and once after it. The layout is: `[response] [user bubble] [response]` instead of `[user bubble] [response]`. Affects all messages after the first.
**Expected:** `[user bubble] [response]` — single response after user bubble.
**Root cause:** The streaming response is appended to the message list as it arrives. When the session ID remaps (85ce37a9 → 27200842), the client fetches history from the new session, which inserts the user message and the response again — but the originally-streamed response is not removed. The result is the streamed copy above the newly-inserted user bubble and the history copy below it.
**Likely location:** `useChatSession` / streaming message state management — the streamed messages are not cleared when the session remap triggers a history reload from `GET /api/sessions/:id/messages`.
**ADR context:** ADR-0117 (direct SSE as sole web client transport) — the POST response body IS the SSE stream; session ID remapping happens within this stream.
**Recommendation:** When the session ID remap event is received (the `session_id_remapped` or equivalent event from the stream), clear the in-flight optimistic message list before the history fetch resolves, so the history-fetched messages are the single source of truth.

### Bug #2 — Model/Permission Mode Selector No-Op Before First Message
**Observed:** Selecting Haiku or any non-default model before sending the first message has no effect — the status bar stays "Sonnet 4.5" and no PATCH request is fired.
**Expected:** User should be able to choose model before the first message.
**Root cause:** `use-session-status.ts:90`: `if (!sessionId) return;` — when no session exists yet (URL has no `?session=` param), `updateSession` exits immediately before setting optimistic state or making the API call.
**Recommendation:** Store the pending model/permission mode selection in a Zustand preference store (keyed to `selectedCwd` or globally) and apply it when the first message creates the session. This pre-selection should be passed as `model` in the initial `POST /sessions/:id/messages` call.

### Bug #3 — Model Display Stale After Model Change (Won't Reflect Selection)
**Observed:** After selecting Haiku via the model dropdown (PATCH 200 confirmed), the status bar continues to display "Sonnet". Selecting permission mode ("Accept Edits") works correctly.
**Root cause:** Priority chain in `use-session-status.ts:69`: `localModel ?? streamingStatus?.model ?? session?.model ?? DEFAULT_MODEL`. The convergence effect (`useEffect` at line 116) correctly clears `localModel` once `session?.model` matches from the PATCH response. But then `streamingStatus?.model` (`claude-sonnet-4-6` from the last SSE event) has higher priority than `session?.model` and takes over. Permission mode has no equivalent `streamingStatus?.permissionMode` field, so it works correctly.
**Recommendation:** After streaming ends (`isStreaming` becomes false), stop using `streamingStatus?.model` in the priority chain. Change to: `localModel ?? (isStreaming ? streamingStatus?.model : null) ?? session?.model ?? DEFAULT_MODEL`.

### UX Issue #1 — First Message Appears Without Bubble During Session Remap
**Observed:** When a new session is started and the first message is sent, the user's text appears as plain unformatted text (no bubble) briefly before flashing into the proper bubble after the session ID remap resolves.
**Expected:** The user message should render in a consistent bubble from the moment it is submitted.
**Root cause:** The client pre-generates a UUID for the URL (`?session=85ce37a9`). The server processes the message and creates an SDK session with a different UUID (`27200842`). During this remap window, the client doesn't know the final session ID, so the user message may render outside the normal message bubble component.
**Recommendation:** Show the user's message immediately in its full bubble styling using optimistic state, independent of session ID resolution. The bubble should be stable; only the session URL param should update when the remap fires.

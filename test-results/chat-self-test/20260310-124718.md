## Summary

The DorkOS chat UI is functional with good code rendering, streaming, and history reload. However, **2 bugs** and **2 UX issues** were identified. The most critical bug is a race condition in the message history seeding logic that causes user messages to be dropped during live streaming on the Relay transport.

## Issues Found

### 1. User Message Dropped During Live Streaming — Bug (P0)

**Observed:** Message 3's user bubble ("Write a minimal HTML page...") was not rendered in the DOM during live streaming. The assistant's response appeared directly after the previous assistant message. After a page reload, the user message appeared correctly.

**Expected:** All user messages should be visible immediately after submission, both during live streaming and after history reload.

**Root cause:** Race condition in `apps/client/src/layers/features/chat/model/use-chat-session.ts` lines 195-225. When a new message triggers a sessionId change (create-on-first-message pattern via Relay), the history reset effect at line 199 unconditionally resets `historySeededRef.current = false`. If the history query refetches before the optimistic message is persisted to disk, the seed effect at line 211 replaces the entire messages array with server history, losing the optimistic user message.

The guard at line 200 (`if (statusRef.current !== 'streaming')`) only protects the `setMessages([])` call but NOT the `historySeededRef` reset. The seed path at line 211 has no streaming guard.

**ADR context:** ADR-0104 (client-side message queue with auto-flush) — the queue itself works correctly; the bug is in the history reconciliation.

**Recommendation:** Guard the `historySeededRef` reset with a streaming check, AND add ID-based deduplication in the seed effect (similar to the polling update path at lines 217-223) for defense-in-depth.

---

### 2. Model Selector Dropdown Doesn't Visually Update — Bug (P1)

**Observed:** Clicking a model option in the dropdown (e.g., Haiku) closes the dropdown but the status bar continues showing the previous model. `aria-checked="false"` on ALL radio items. Server-side model change DOES eventually apply.

**Expected:** Status bar should immediately reflect the selected model, and the dropdown should show the active model as checked.

**Root cause:** In `apps/client/src/layers/entities/session/model/use-session-status.ts` lines 77-100, the `updateSession` callback applies an optimistic `localModel` state, then clears it after the PATCH succeeds. The priority chain at line 59 (`localModel ?? streamingStatus?.model ?? session?.model ?? DEFAULT_MODEL`) falls through to stale values when `localModel` is cleared and the TanStack Query cache update doesn't properly propagate.

The `aria-checked="false"` issue occurs because the Radix RadioGroup's `value` prop doesn't match any rendered item value (string format mismatch between the resolved model value and the model option values).

**Files:** `ModelItem.tsx` (line 38), `use-session-status.ts` (lines 59, 77-100), `StatusLine.tsx` (lines 88-92).

**Recommendation:** Ensure the query cache is invalidated/refetched after PATCH success rather than relying solely on optimistic clearing. Fix the RadioGroup value to match item values exactly.

### 3. Excessive API Polling — UX Issue (P2)

**Observed:** During a single message exchange, `pulse/runs` and `git/status` were polled 5+ times each within ~15 seconds. 111 Vite HMR connection messages logged at page load.

**Expected:** Polling should be less aggressive, especially for `git/status` which is unlikely to change during a chat message.

**Recommendation:** Increase polling intervals for `pulse/runs` and `git/status` during active streaming, or pause non-essential polling while a message is in flight.

### 4. Tool Use Block Renders as Empty Message — UX Issue (P2)

**Observed:** After history reload, the TodoWrite `tool_use` assistant block renders as a separate message item at DOM index 7 with no visible content (just a timestamp "12:58 PM"). The text response follows as a separate message at index 8.

**Expected:** Tool use blocks should either be collapsed/hidden or rendered as a tool call card (expandable).

**Recommendation:** Verify that the transcript parser correctly groups tool_use and tool_result blocks with their surrounding text messages, or render tool_use blocks with a meaningful card UI.

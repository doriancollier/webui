## Issues Found

### Issue 1: 503 Responses on Old Sessions (Pre-Existing, Fixed in 23afdbb) — ✅ VERIFIED FIXED
**Observed:** Network log showed 30+ repeated 503 responses for `GET /api/sessions/b4e077dd.../messages` and `/stream` on old sessions during initial page load.

**Root cause (researched):** A relay-mode polling storm bug — `useChatSession`'s `refetchInterval` lacked a `relayEnabled` guard, causing 3-second polling of `GET /messages` even when SSE already handled history sync. This exhausted SSE connection limits in `session-broadcaster.ts` (`MAX_CLIENTS_PER_SESSION: 10`), returning 503.

**Fix location:** `apps/client/src/layers/features/chat/model/use-chat-session.ts` — commit `23afdbb` added `|| relayEnabled` guard to `refetchInterval`.

**Current status:** ✅ Fixed and verified — the **current test session (5b1d562a) has zero 503s**. The 503s in the network log are from pre-test browsing of old sessions before the fix was applied to those open tabs.

**Verdict:** No action needed. Fix is working.

### Issue 2: Tool Result "Done" Text Visible Outside Collapsed Card During Streaming — UX Issue
**Observed:** During live streaming of Message 4 (TodoWrite), the text "Done" appeared as a standalone element between the collapsed tool card and the assistant's text response. In history view, this text does not appear.

**Expected:** The tool result should either be hidden inside the collapsed card (consistent with history) or shown as part of the card's preview — not as a floating text node between card and assistant message.

**Root cause (researched):** The `ToolCallCard.tsx` only renders result text when `expanded === true` (inside `AnimatePresence`). The standalone "Done" text is NOT from ToolCallCard. It's likely being rendered from a different source during streaming — possibly the `tool_result` SSE event content being surfaced as a separate text element, or the JSONL line 24 (user message with tool_result content: "Done") being partially rendered during the live streaming/sync path.

**Affected file:** `apps/client/src/layers/features/chat/ui/MessageItem.tsx` or `stream-event-handler.ts` — exact line not confirmed.

**Recommendation:** Investigate how the `tool_result` SSE event is rendered in the message list. The "Done" text should not be visible as a standalone element — it should be contained within the collapsed/expanded tool card only. The inconsistency between streaming and history view suggests the streaming path has an extra render path that leaks tool result content.

### Issue 3: Auto-Scroll Disengages After Short Message Following Long Response — UX Issue
**Observed:** After sending Message 5 ("What is 2+2?"), streaming completed in ~14s but the viewport did not scroll to show the new response. The ↓ button appeared, requiring a manual click.

**Expected:** After streaming completes, the chat panel should auto-scroll to the bottom to show the newest message.

**Root cause (researched):** `MessageList.tsx` controls auto-scroll via `isAtBottomRef` with a 200px threshold (line ~109). Auto-scroll only fires when `isAtBottomRef.current === true`. During the Message 4 → Message 5 transition:
1. Message 4 was very long (tool call + task list + explanation)
2. After Message 4 completion, clicking the input area may have triggered a layout reflow changing `scrollHeight`
3. If this shifted `isAtBottomRef.current` to `false`, subsequent messages wouldn't auto-scroll

Auto-scroll is guarded by `ResizeObserver` + a fallback `useEffect` on `messages.length` — both check `isAtBottomRef.current`. If that ref is false due to the layout shift from a long response, auto-scroll silently stops working until the user scrolls to the bottom and re-enables it.

**Affected file:** `apps/client/src/layers/features/chat/ui/MessageList.tsx` lines ~104-200.

**Recommendation:** Consider increasing the threshold from 200px to a larger value (e.g., 400px), or adding a specific handler for "after streaming completes" that forces scroll to bottom regardless of `isAtBottomRef` state. Users expect new AI responses to always be visible.
